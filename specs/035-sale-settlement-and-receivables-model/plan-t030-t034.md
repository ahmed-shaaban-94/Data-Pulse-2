## Summary

Five slices implement the settlement runtime over the merged schema, each conforming
to the ratified contract. The build reuses the repo's established NestJS feature-module
pattern (the `erpnext-warehouse-map` + `sale-sync-ops` precedents): a `*.module.ts`
registered in `app.module.ts`, a `*.controller.ts` (guards + Zod DTO validation +
discriminated-union→HTTP mapping), a `*.service.ts` (one `runWithTenantContext`
transaction per write, version-optimistic UPDATEs, `23505/23503` Pg-code catch), and
`dto/`. The one POS route uses the operator-envelope guard; all Console routes use the
cookie/session stack. The reconciliation worker follows the `apps/worker` processor
precedent.

## Architecture conventions (reused, not invented)

| Concern | Repo pattern to follow | Precedent |
|---------|------------------------|-----------|
| Module shape | `module / controller / service / dto`, imported in `app.module.ts` | `erpnext-warehouse-map`, `sale-sync-ops` |
| Tenant scope | one `runWithTenantContext(tenantId, storeId, tx => …)` per write/read | warehouse-map.service |
| Service results | discriminated union `{kind:"ok"|"conflict"|"not_found", …}`; **service never throws HttpException** — controller maps to 200/201/404/409 | warehouse-map.service |
| Optimistic concurrency | `UPDATE … WHERE id=$1 AND version=$2 …`; 0-row ⇒ conflict/not_found; version++ | warehouse-map.retire |
| Unique/FK conflicts | catch Pg `23505` ⇒ conflict, `23503` ⇒ not_found/conflict | warehouse-map.service |
| Console auth | `@UseGuards(DashboardAuthGuard, TenantContextGuard)` + per-route `RolesGuard`/`@Roles` | warehouse-map.controller |
| POS auth | the operator-authorization envelope guard (031) on the one POS route | pos-sales sale-write guard |
| Idempotency | existing `IdempotencyInterceptor` + `Idempotency-Key` header (contract `x-idempotency: required`) | sale-sync-ops repair |
| Audit | `@Auditable` + `AuditEmitterInterceptor`; or `AuditJobEnqueuer` in-tx | warehouse-map |
| Validation | `ZodValidationPipe` on `@Body`/`@Param`, schemas mirroring the contract | every controller |
| Worker | `apps/worker/src/settlement/*.processor.ts`, registered in `worker.module.ts` | erpnext-reconciliation processor |
| Cross-tenant | safe-404 (RLS returns 0 rows ⇒ not_found), never 403 | §II/§XII |

## Slices

### T030 — Receivable lifecycle + settlement-intent capture  *(P1, foundation)*
**Module:** `apps/api/src/settlement/` (`settlement.module.ts`, `settlement.controller.ts`,
`receivable.service.ts`, `dto/`).
**Routes (from contract):**
- `posRecordSettlementIntent` (POS, operator-envelope guard) — opens receivable(s)
  from a captured sale + payer split; idempotent; unknown/cross-tenant payer ⇒ 409
  unknown-payer (safe). Writes N receivable rows in one tx; does **not** mutate the sale.
- `consoleGetReceivable` / `consoleListReceivables` (Console) — projections; keyset
  pagination; object-level safe-404.
**State machine:** `open → partially_applied → settled`; `open → claimed`; variance/
net-zero ⇒ `flagged`. Encoded as guarded transitions in the service (no `reversal_consumed`).
**Tests:** service unit (transition table + replay-safety), controller contract
(routes/DTO/status), RLS isolation (cross-tenant ⇒ 404). RED-first.
**Guards touched:** none of the frozen exhaustiveness guards (no migration, no new
outbox event/metric here).

### T031 — Cash application (7-C, DP-2-owned)  *(P2)*
**Route:** `consoleApplyPayment` (Console). Append a `payment_application` row + reduce
`receivable.outstanding_balance` in one `runWithTenantContext` tx, version-guarded.
**Outcomes (contract):** clearing ⇒ `settled`; partial ⇒ `partially_applied`;
over-application (amount > balance) ⇒ **409 conflict** (never silent truncation); stale
version ⇒ 409; idempotent replay never double-reduces.
**7-C:** writes only the DP-2 operational record; `erpnext_payment_entry_ref` stays null
(no posting — connector-owned, gated).
**Tests:** partial/clearing/over-application/replay; audit record asserted.

### T032 — Claim & remittance reconciliation  *(P3, worker)*
**API routes (Console):** `consoleSubmitClaim` (creates `claim` + `claim_receivables`,
transitions receivables → `claimed`); `consoleReconcileRemittance` (records `remittance`
+ `reconciliation_result` with variance).
**Worker:** `apps/worker/src/settlement/reconciliation.processor.ts` — async
reconciliation per the erpnext-reconciliation processor precedent; settles/flags
receivables from remittance.
**Reuse, not reinvent:** claim-line rejection routes to DP-026 + Connector Arc A +
POS-014 — **no rejection/reversal op authored** (NG-1).
**Outcomes:** full ⇒ settled/variance 0; partial ⇒ balance + variance; net-zero-or-below
⇒ flagged; claim not in reconcilable state ⇒ 409; out-of-order remittance ⇒ defined hold.
**Guards touched:** **IF** the worker emits a settlement outbox event (e.g.
`settlement.reconciled`), it MUST be appended to `event-types-registry.spec`
`EXPECTED_EVENT_TYPES` (the #534-class catch) — decide at slice time whether
reconciliation is event-driven; if not, no guard change.

### T033 — Authorization wiring + isolation verification  *(cross-cutting)*
Not a standalone module — the guard stacks are applied in T030–T032 controllers
(Console = `DashboardAuthGuard+TenantContextGuard+RolesGuard`; POS = operator envelope).
This slice is the **cross-cutting verification**: malicious-override + cross-tenant
safe-404 tests across every settlement route, wrong-surface-credential rejection (cookie
on POS route / envelope on Console route ⇒ 401), and confirmation that mid-session
operator revocation stops settlement writes (the 031 live re-verification). Required by
Principle VI (server-side authorization tests).

### T034 — Observability signals  *(polish)*
Settlement/receivable health metrics (per spec §7): e.g. open-receivable count/age,
cash-application throughput, reconciliation variance rate, replay-suppression count.
**Guard touched (HARD):** every new metric name MUST be added to
`apps/api/test/observability/cardinality.spec.ts` `expectedSignals` **and** the shared
`ALLOWED_METRIC_LABELS` (the #447/#534-class frozen-set guard) — run the worker-signals
suite too if a worker metric is added. This is the one slice that definitely touches a
frozen guard.

## Dependency order

```
T030 (receivable + intent) ─► T031 (cash application) ─► T032 (claims/recon worker)
        │                            │                          │
        └──────────── T033 (authz + isolation verification, spans all) ───────────┘
                                     └──► T034 (observability, after the ops exist)
```
T030 is the foundation (everything writes receivables). T031 needs receivables to apply
against. T032 needs receivables to claim. T033 verifies all. T034 instruments all.

## Test strategy (per slice, TDD RED-first)

- **Service unit** — transition tables, money math, replay-safety, conflict paths
  (no HTTP, no DB mock beyond the tx wrapper seam).
- **Controller contract** — routes/DTOs/status codes match `settlement.yaml`
  (the per-feature `*.contract.spec.ts` precedent).
- **Integration (WSL Testcontainers)** — RLS isolation, real tx, version conflicts,
  idempotent replay. **`db-integration` is the authoritative gate** for any slice that
  touches the DB (the T020 lesson) — verify the round-trip/integration locally in WSL
  before push; do not merge a DB-touching slice on `fast`-green alone.
- **Coverage** — 80%+ (repo policy); malicious-override + cross-tenant tests required.

## Guards & forbidden-path checklist (per slice)

- `apps/**` is not a `[GATED]` path — but each slice is still its own branch + PR.
- **No** schema change (schema shipped in 0027). If a slice discovers a missing
  column/index, that is a NEW gated migration (0028+), not an edit to 0027.
- **cardinality.spec** (T034) and **event-types-registry.spec** (T032 if event-driven)
  are the frozen guards in play — update in lockstep or CI breaks.
- `package.json` / lockfile / CI / OpenAPI / `.github` remain untouched (the contract
  is already authored; runtime conforms to it, does not change it).
